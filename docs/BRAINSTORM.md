# BRAINSTORM: ccguard MVP

> Фаза 1 lifecycle. Цель — пройти по развилкам ТЗ, зафиксировать
> рассмотренные альтернативы и выбрать (с обоснованием). Решения отсюда
> уходят в `SPEC.md` без повторного обсуждения.

Статус решений: 🟢 принято / 🟡 на обсуждение с пользователем / 🔴
отложено за скоуп MVP.

## 1. Механизм enforcement: только хуки?

**Альтернативы:**

| Подход | Плюсы | Минусы |
|---|---|---|
| **Хуки Claude Code (`PreToolUse`)** | Штатный механизм, не нужен root, кросс-платформа, нативно знает `tool_name`/`tool_input`, отдаёт allow/deny по протоколу | Защищает только то, что Claude Code сам вызывает. Не ловит, что MCP-сервер делает по сети. Можно отключить (`disableAllHooks` или удалив конфиг). |
| LD_PRELOAD / DYLD_INSERT | Перехват любых `connect()`/`execve()` | Linux/macOS-only, требует обёртки запуска, ломается при статической линковке, конфликтует с антивирусами |
| eBPF / seccomp | Полный контроль syscalls | Требует root + LSM, не работает в контейнерах для пользователя, не кросс-платформа |
| MITM HTTPS-прокси | Контроль всего сетевого трафика Claude Code и MCP | Нужен CA в trust store, ломает TLS-pinning, инвазивно, отдельный продукт |

**Решение:** 🟢 хуки. ТЗ это и предписывает. OS-уровень — отдельная
итерация, флаговать в README как «known limitation: agent can be bypassed
by removing the hook».

**Дополнительно:** ccguard должен периодически (`ccguard check`)
верифицировать, что его хук всё ещё установлен и не модифицирован
(сравнение с эталоном). Tamper-detection как компенсация.

## 2. Открытый вопрос #2: видимость MCP/сети из хука

Из [HOOKS_PROTOCOL.md](HOOKS_PROTOCOL.md) §5: PreToolUse получает
`tool_name`/`tool_input`. Для MCP это значит:

- Видим **имя MCP-инструмента** (`mcp__github__create_issue`) и его
  аргументы — фильтровать по имени и по содержимому полей можно.
- НЕ видим, к каким URL/хостам обращается MCP-сервер изнутри.
- Для `WebFetch`/`WebSearch` URL виден в `tool_input.url` — можно
  проверять `network` denylist.

**Решение:** 🟢 двухконтурная модель в SPEC:

- **Runtime (enforce):** Bash (по regex), MCP-вызовы (по имени тула и
  matcher `mcp__<server>__.*`), WebFetch/WebSearch (по host).
- **Static (check):** все остальные параметры `network`/`mcp_servers`
  policy — анализ конфига MCP-серверов (`command`, `args`, `env`,
  `url`). Findings уровня `block` поднимаются на этапе scan, до запуска.

## 3. Открытый вопрос #3: версионирование политики

**Альтернативы:**

| Механизм | Плюсы | Минусы |
|---|---|---|
| Поле `version: int` в policy + `If-None-Match: <version>` от агента | Простота, понятная семантика, легко логировать | Изобретаем ETag своими руками |
| HTTP `ETag` (хэш контента) + `If-None-Match` | Стандарт HTTP, поддерживается из коробки FastAPI | Менее очевиден агенту, что версия «новее» (хэши не сравнить) |
| Постоянный pull без кэша | Нулевая логика | Нагрузка на сервер, латентность |

**Решение:** 🟢 **гибрид**. В policy.yaml есть поле `schema_version` (для
обратной совместимости формата) **и** `revision: int` (монотонно растущее
число — версия содержимого). Сервер выставляет `ETag: "rev-<revision>"`,
агент шлёт `If-None-Match`. При совпадении — `304`, агент остаётся на
кэше. Не совпало — забирает новую policy.

## 4. `machine_id`: стабильность vs анонимность

**Альтернативы:**

- `/etc/machine-id` (Linux) / `IOPlatformUUID` (macOS) — стабильный, но
  идентифицирует машину.
- hostname — нестабилен, PII (часто содержит имя сотрудника).
- MAC-адрес — нестабилен в виртуалках, считается PII.
- Псевдоним из конфига агента — пусто из коробки, ничего не идентифицирует.

**Решение:** 🟢 `sha256(machine-id || user-uid || install-salt)`,
обрезанное до 128 бит и закодированное base32. Соль генерируется при
`ccguard install` и хранится в `~/.ccguard/config.yaml`. Сервер видит
стабильный псевдоним, не имеющий обратимой связи с хостом. В `config.yaml`
оставить опциональное поле `machine_label` — человекочитаемая метка,
если организация хочет (например, `"laptop-anton"`).

## 5. `fail_open` по умолчанию: безопасно ли?

ТЗ предлагает `fail_open: true` по умолчанию, «чтобы не ломать
разработчика». Это противоречит security-философии проекта.

**Альтернативы:**

- `fail_open: true` глобально — мягко, но дыра.
- `fail_open: false` глобально — строго, но один проёб в policy ложит
  всем разработчикам Claude Code.
- **Дифференциация по severity:** `info`/`warn` правила всегда
  fail-open; `block` правила имеют отдельную настройку
  `block_fail_mode: open|closed`, default — `open`.

**Решение:** 🟢 третий вариант. По умолчанию `block_fail_mode: open`
(как в ТЗ), но видимая настройка с явной семантикой, чтобы организация
могла включить closed. Каждое fail-open срабатывание пишется в audit-лог
с пометкой `fail_open_triggered`.

## 6. Audit-лог: только локально или дублировать на сервер?

ТЗ говорит: локальный `~/.ccguard/audit.log` (JSON-lines).

**Соображение:** аудит локально = тривиально удалить = бесполезен для
расследования. Но отправка каждого hook-вызова в сервер = нагрузка и
latency.

**Решение:** 🟢 локально пишем всё, на сервер шлём **только deny-решения
и fail_open-события** batch'ем в составе следующего `ccguard sync`. Allow
не шлём (объём + бесполезно для security-сигнала). Для MVP — на этом
останавливаемся; стриминговый аудит — за скоупом.

## 7. Хэширование скиллов: только SKILL.md или включая скрипты?

`SKILL.md` — это инструкция; скрипты в той же папке могут быть
подменены без изменения `SKILL.md`. Если хэшировать только md — security
gap.

**Решение:** 🟢 хэшируем **папку скилла целиком** (sha256 от
отсортированного списка `path: sha256(content)` всех файлов в директории).
Это и есть подлинный fingerprint. Поле в inventory: `dir_hash`. Поле в
policy `trusted_hashes` — список доверенных `dir_hash`.

Поле `signature` в policy — заглушка под Sigstore/cosign-подпись в
будущей итерации, 🔴 не реализуем в MVP.

## 8. Аутентификация агентов на сервере

ТЗ: статический API-токен в заголовке `X-CCGuard-Token`. Bootstrap —
слабое место (как токен попадает на 100 машин?).

**Решение:** 🟢 для MVP — статический токен, как в ТЗ. В config.yaml
сервера — список валидных токенов с метками: `{token: "...", label:
"team-frontend"}`. Это уже позволяет ротировать частично. mTLS,
per-machine токены через registration flow — 🔴 за скоупом.

Один nuance: токен **не должен** попадать в логи сервера и в ответы API
(маска `***`).

## 9. Хранилище сервера: SQLite vs Postgres

ТЗ: SQLite. Лимит ясен — конкурентные writes плохо, при сотнях агентов
упирается. Для MVP (демо/портфолио/малый пилот) хватит.

**Решение:** 🟢 SQLite через **SQLModel** (pydantic-friendly), WAL-mode.
Слой репозитория абстрагирует БД — миграция на Postgres = смена URL
SQLAlchemy без переписки доменной логики. В README — секция «when to
outgrow SQLite».

## 10. Куда `ccguard install` пишет хук

**Альтернативы:**

- `~/.claude/settings.json` — пользовательский, покрывает все проекты.
- `.claude/settings.json` — проектный, шарится через VCS.
- managed settings — системный, требует root, для централизованного
  enforcement идеален.

**Решение:** 🟢 default — пользовательский (`~/.claude/settings.json`),
флаг `--scope=project|user|managed`. Managed — пишем только если процесс
имеет соответствующие права и каталог существует, иначе fallback на user
с предупреждением.

## 11. Что именно вызывает Claude Code как хук

**Альтернативы:**

- Напрямую `ccguard enforce` — требует `ccguard` в `$PATH` Claude Code.
- Shim-скрипт в `~/.ccguard/bin/ccguard-enforce` (генерим при install) —
  абсолютный путь в settings.json, не зависит от `$PATH`.

**Решение:** 🟢 shim. Абсолютный путь в settings.json устойчивее, плюс
shim может выполнять диагностику (если ccguard сломан — fail-open с
понятной ошибкой в stderr). Shim — bash-скрипт на ~20 строк.

## 12. Что попадает (и НЕ попадает) на сервер

**Не отправляем никогда:**

- Значения env-переменных MCP-серверов (только имена ключей).
- Содержимое команд из audit-лога (только regex/правило, по которому
  заблокировали, и tool_name).
- `transcript_path` из hook payload.
- Содержимое скиллов (только хэши).

**Отправляем:**

- Метаданные конфигов (откуда взяты, какие хуки, какие MCP-серверы по
  именам и типам).
- Findings (id правила, severity, source, recommendation).
- Аудит блокировок (только deny + fail_open).
- `machine_id` (псевдоним, см. §4) + `machine_label` (если задан).

**Решение:** 🟢 зафиксировать в SPEC раздел «Data minimization» с
референс-листом полей.

## 13. Reflexion: что в этом списке может быть неправильно?

- **§4 machine_id:** если организация использует одинаковые VM-образы
  (`/etc/machine-id` совпадает), наш псевдоним перестаёт быть уникальным
  → сервер сольёт findings нескольких машин. Контрмера: добавить в
  hashed-источник `hostname || install-salt` — install-salt уникален на
  каждой инсталляции. Уже учтено выше, но подсветить в тестах.
- **§3 версионирование:** если у организации **несколько политик**
  (например, по командам), revision-int перестаёт работать
  (revision у команды A не сравним с команды B). Для MVP считаем одну
  политику на инсталляцию сервера. Multi-tenancy — 🔴 за скоупом, явно
  записать в README.
- **§11 shim:** если ccguard переустановлен в другое место, shim
  ссылается на старый путь → fail-open молча. Контрмера: `ccguard
  install` идемпотентен и при каждом запуске обновляет shim.
- **§2 двухконтурная модель:** разработчик может обойти static-check,
  собрав MCP-сервер с динамическим списком хостов из env. Это known
  limit, фиксируем в README.

## Финальные решения после интерактивного review

Подтверждено пользователем (2026-05-23):

| § | Решение |
|---|---|
| §2 | Двухконтурная модель: runtime + static |
| §3 | `revision: int` в policy + HTTP `ETag`/`If-None-Match` |
| §4 | `sha256(machine-id ‖ uid ‖ install-salt)` + опциональный `machine_label` |
| §5 | Дифференциация по severity, `block_fail_mode: open|closed`, default `open` |
| §6 | На сервер шлём только `deny` и `fail_open`-события батчем в `sync` |
| §7 | `dir_hash` всей папки скилла |
| §10 | Default install = user (`~/.claude/settings.json`), флаг `--scope=project|user|managed` |
| §11 | Shim-скрипт в `~/.ccguard/bin/ccguard-enforce` |
| §13 | Tamper-detection: сравнение `settings.json` с эталоном (хук присутствует, ведёт на наш shim, `disableAllHooks=false`) |
| §14 | Performance: `enforce` собираем в бинарник PyInstaller'ом — startup <30мс |
| §15 | Audit-лог: `RotatingFileHandler`, 10MB × 5 файлов |

### Финальный enforce-scope (runtime через хуки)

- ✅ `Bash` — против `commands` denylist (regex/glob).
- ✅ `mcp__*` — против `mcp_servers` denylist (по имени сервера и
  конкретного tool'а).
- ✅ `WebFetch` / `WebSearch` — URL из `tool_input` против `network`
  denylist/allowlist.
- ❌ `Edit` / `Write` / `Read` — **за скоупом MVP**. Секция `policy.paths`
  не вводится. Можно вернуть в v2, если будут реальные кейсы.

Соответственно `policy.example.yaml` имеет четыре доменных секции:
`mcp_servers`, `network`, `commands`, `skills`, `hooks`. Без `paths`.

### Изменения, требующие правки исходных §§

- §10 →  ещё актуально (scope flag).
- §12 (data minimization) → актуально без изменений.
- §13–§15 — добавлены по итогам review (tamper / perf / log rotation).
