Как это работает

Один проход APScheduler-тика: anomaly → risk → sequence. Один writer, общая сессия, per-machine try/except — одна плохая машина не валит весь sweep.

APScheduler tick ↓ agent: signal extractor (catalog regex per PostToolUse event) ↓ tags only (privacy: no raw tool_input ever leaves endpoint) server: ToolUseEvent.signals_json ↓ ├─ anomaly_tick — 3σ baseline (legacy, kept running) ├─ risk_tick — Σ weight·2^(-age/half_life) > threshold → "risk.elevated" [warn] └─ sequence_tick — cred.read.* → egress.* in <15min → "ioa.exfil_sequence" [high] ↓ FindingRecord ↓ UI: machine_detail expandable explainability + fleet risk panel

slide 4 (bonus) · rule discovery loop

Самообучающийся каталог сигналов NEW

Цикл, который позволяет добавлять новые behavioral signals без code-change и redeploy. Новая ATT&CK техника выходит в понедельник — у нас в среду она уже ловится regex'ом на всех агентах.

02:00 UTC — APScheduler tick (once per day) ↓ source_monitors (3 источника, stdlib urllib, fail-isolation) • AtomicRedTeamMonitor — GitHub releases redcanaryco/atomic-red-team • MitreAttackMonitor — mitre/cti releases • CVEAIFilterMonitor — NVD CVE feed, фильтр по AI/agent keywords ↓ new SourceItem (URL ещё не в SourceFetchLog) signal_drafter → Anthropic API (Claude Sonnet 4.6) • Few-shot: 4 примера из реального CATALOG • Budget gate: общий с LLM scanner (daily_call_budget) • Markdown-fence recovery для упрямых LLM ↓ draft JSON ProposedSignal (status=pending) → SourceFetchLog (dedup на след. дни) ↓ 09:00 UTC — админ на /admin/proposed-signals • Видит draft + source link + LLM rationale + pattern • [Approve] валидирует regex.compile() → SettingsRecord["catalog.override.<id>"] ↓ /api/v1/policy (E4) injects signal_overrides + ETag bump ↓ agent sync подхватывает policy.yaml (ETag invalidation) ↓ hook hot path (<20ms) → extract_signals(..., overrides=load_overrides(...)) • Override id == baked id → override REPLACES (admin hotfix FP regex) • Новые id → дополняют CATALOG ↓ 🎯 Новый сигнал ловится на всех агентах без redeploy

Privacy

LLM видит только public threat-intel. Customer telemetry никогда не пересекает границу.

Human-in-the-loop

Без approve клика — никакой сигнал не активируется. LLM hallucination остаётся в очереди.

Cost gate

Тот же daily_call_budget, что у scanner. Один knob контролирует всю LLM стоимость.

slide 4 · design invariants

Архитектурные принципы

Privacy by construction

Сервер никогда не видит сырой tool_input. Агент извлекает tags по regex-каталогу и шлёт только ID + fingerprint. Все finding payload'ы — IDs + числа.

Forward-compatible

Старый агент (v0.1) шлёт signals_json = "[]" — kernel выдаёт score 0, finding не выпускается. Неизвестный signal ID получает weight 1.0, а не exception.

Tunable без redeploy

Все пороги в SettingsRecord: risk.threshold, risk.window_hours, risk.half_life_hours, sequence.window_minutes, enforcement_mode. Idempotent seeders, admin-edits preserved.

Explainability built-in

FindingRecord.payload_json несёт breakdown: score / threshold / contributions / event_count. UI рендерит без дополнительных запросов. Каждый signal → ATT&CK URL.

Среднесрок

  • Peer-group normalization — нормировать score по похожим машинам, не по абсолюту (snowflake-resistant)
  • Signal coverage: ещё ~20 IDs (data-staging, exfil-over-cloud-storage, container-escape attempts)
  • Webhook/Slack alert delivery вместо HTMX dashboard polling
  • Multi-tenant (v0.3 milestone, упомянуто в CLAUDE.md)

Долгосрок

  • ML-baselining light: clustering machines по signal-distribution, не VAE (избегаем "heavy ML" по спеке §7)
  • Native SIEM bridge (Splunk HEC, Elastic, Datadog) — push FindingRecord как ECS-compliant event
  • Postgres migration > 100 машин (планировалось в v0.3)

Технический долг

  • Brittle exact-markup тесты в render layer — нужна стратегия снапшот-тестирования
  • SQLite tz-naive datetimes: gotcha повторяется в каждом новом сервисе (risk_service, sequence_service) — вынести helper
