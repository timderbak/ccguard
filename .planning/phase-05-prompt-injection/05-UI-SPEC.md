## Information Architecture

### Navigation

**No new sidebar entries. No new tab.** The Prompt-Injection editor lives inside the existing `/policy` page (Правила tab) as one additional `<details>` card appended after the existing 7 section cards.

Final sidebar order: `Обзор / Машины / Находки / Аудит / Аномалии / Политика / История / Настройки` — unchanged from Phase 4.

### Page structure: `/policy` (extension — one new section card)

The existing `policy_editor.html` form contains 7 `<details>` cards. Phase 5 appends an **8th card** at the bottom of the form, immediately before the sticky action bar:

```
/policy form (Правила tab)
├─ <details> MCP                       (existing)
├─ <details> Network                   (existing)
├─ <details> Commands                  (existing)
├─ <details> Skills                    (existing)

├─ <details> Agents                    (existing)
├─ <details> Env                       (existing)
├─ <details> Prompt-Injection          (NEW)
```

The card is rendered via a new partial `components/_policy_section_prompt_injection.html`, included after `_policy_section_env.html` in `policy_editor.html`.

### Card layout (top-to-bottom inside `<details>`)

1. **Card heading** — `<summary>Prompt-Injection</summary>`
2. **Enabled toggle row** — checkbox + label `Включить детекцию prompt-injection`
3. **Severity dropdown row** — `<select>` with 3 options (`info` / `warn` / `block`)
4. **Regex patterns textarea** — line-per-pattern, with helper text
5. **Allowlist patterns textarea** — line-per-pattern, with helper text
6. **LlamaGuard sub-block** (nested `<fieldset>` with border):
   1. Enabled toggle (`Включить LlamaGuard (deep-scan)`)
   2. Endpoint URL input
   3. Timeout (ms) input

No add/remove row buttons — textareas hold the lists (one pattern per line). This deliberately differs from Phase 4 mandatory rows (which use HTMX add/remove for typed records) because regex patterns are plain strings with no schema.

## Interaction & State

### Editing

- All controls are plain HTML form elements. **No HTMX**, no inline JS in this card (the entire card re-renders on standard form POST).
- Changes are not persisted until the user clicks «Сохранить черновик» in the sticky bottom bar (shared with the rest of the form).
- Disabling the top-level «Включить детекцию» toggle does **not** visually disable the other fields (server-side ignores them when `enabled=False`; UI keeps them editable so the admin can stage config without losing values). This matches the v0.1 pattern for `mcp_servers.enforce`.

### Saving draft

- Single combined `<form method="POST" action="/policy/draft">` — same form that wraps the other 7 sections. No separate submit for this card.
- Server validates `prompt_injection` section with Pydantic v2:
  - Each line in `regex_patterns` and `allowlist_patterns` must compile via `re.compile(...)` — invalid patterns rejected.
  - `severity` must be one of `info` / `warn` / `block`.
  - `llama_guard.endpoint` must parse as a URL (http:// or https://) when `llama_guard.enabled=True`.
  - `llama_guard.timeout_ms` must be integer in `[50, 10000]`.
- **Validation error path:** re-render `/policy` with an error notice atop the Prompt-Injection card (red `<p>` inside the `<details>` block, above the toggle row); draft content preserved in form values; the offending line number(s) cited where possible (e.g. `Невалидный regex в строке 3: «\K(?=foo)»`).
- **Success path:** 303 redirect to `/policy` (existing behavior); `has_draft` flips to `True`; revision header shows `· не сохранено`.

### Publishing

Identical to existing flow: `<button type="submit" formaction="/policy/publish">Опубликовать</button>` with `onclick="return confirm('Опубликовать черновик на все машины?')"`. Unchanged.

### States

| State | Trigger | UI |
|-------|---------|-----|
| Default (with draft, enabled) | `has_draft=True`, `prompt_injection.enabled=True` | Card `<details open>`; toggle checked; severity reflects draft; textareas populated; LlamaGuard sub-block reflects draft |
| Default (no draft, never configured) | first load after Phase 5 deploy, no policy section yet | Card open; toggle **checked** (default-on per CONTEXT.md `enabled: bool = True`); severity `warn` (default); regex_patterns empty (engine uses built-in default set); allowlist empty; LlamaGuard sub-block: enabled unchecked, endpoint `http://localhost:11434`, timeout `500` |
| Disabled scanner | `prompt_injection.enabled=False` | Toggle unchecked; all other fields still editable (greyed not applied — values preserved) |
| LlamaGuard disabled | `prompt_injection.llama_guard.enabled=False` | LlamaGuard toggle unchecked; endpoint + timeout inputs still editable; helper text reads `Опционально — deep-scan через локальный Ollama. По умолчанию выключен.` |
| Validation error — invalid regex | one or more lines in `regex_patterns` fail to compile | Re-render `/policy`; red notice atop card: `Невалидный regex в строке {N}: «{pattern}». Исправьте и сохраните снова.`; all form values preserved |
| Validation error — bad endpoint URL | `llama_guard.enabled=True` AND `endpoint` not a parseable URL | Red notice: `Endpoint LlamaGuard должен быть валидным URL (http:// или https://).` |
| Validation error — timeout out of range | `timeout_ms < 50` or `> 10000` | Red notice: `timeout_ms должен быть в диапазоне 50–10000 мс.` |
| Empty regex_patterns textarea | textarea blank | Helper text under textarea reads `Пусто — используется встроенный набор паттернов.` (informational, not an error) |
| Empty allowlist_patterns textarea | textarea blank | Helper text reads `Пусто — allowlist отключён.` (informational) |
| Loading | initial page render | Server-rendered HTML; card opens immediately; no spinners |

### Confirmations / destructive actions

| Action | Mechanism | Confirmation copy (locked) |
|--------|-----------|----------------------------|
| Publish draft (existing — unchanged) | `onclick="return confirm(...)"` | `Опубликовать черновик на все машины?` |
| Save draft (this section) | none — non-destructive, reversible via rollback in history | — |
| Toggle scanner off | none — non-destructive; engine simply skips on next sync | — |
| Toggle LlamaGuard off | none — non-destructive | — |
| Switch severity `warn` → `block` | none at form level; the existing publish confirmation copy already warns about cluster-wide effect | — |

**Note:** Switching severity to `block` is the most operationally impactful change in this section (it can deny tool calls cluster-wide). Per CONTEXT.md the actual enforcement happens via pull-based policy fetch on agents; the existing publish confirmation `Опубликовать черновик на все машины?` is the single chokepoint. No additional inline confirmation is added — this matches the existing pattern for changing `mcp_servers.enforce` from `warn` to `block` in v0.1.

|---------|----------------|
| Semantic structure | `<details>`/`<summary>` for collapsible card (matches all 7 existing sections); `<fieldset>`/`<legend>` for LlamaGuard sub-block (group of related controls); `<label>` wrapping every form control with visible text (no placeholder-only labels) |

| Card collapse | `<details>` is natively keyboard-accessible (Enter on `<summary>` toggles). Card defaults to `open`. |
| Color contrast | `text-red-600` for error notices ≥ 4.5:1 on white ✓; `text-slate-400` for informational helpers — 4.5:1 on white ✓ (matches verified Phase 1–4 contrast). |
| Error indication not color-only | Error notices carry full Russian text including the offending line number and the problematic value — text content is the primary signal, red color is reinforcement. |
| Disabled state not color-only | When `enabled=False`, the toggle's unchecked state is the primary signal — fields below are NOT visually disabled (per CONTEXT.md pattern); admin sees the unchecked checkbox as the source of truth. |
| Confirm dialog | Existing publish confirmation (`confirm()`) — browser-native, keyboard-accessible. No new confirmations in this card. |
| Fieldset for LlamaGuard | Native HTML grouping; screen readers announce the legend (`LlamaGuard (опционально)`) when entering any of the 3 child controls. |

## Performance

| Concern | Decision |
|---------|----------|
| `/policy` initial render | One additional `<details>` card. Typical regex_patterns list: 0–30 lines × ~80 chars ≈ 2.4KB. Allowlist similar. LlamaGuard sub-block: ~600 bytes. Total: < 5KB additional payload. Negligible. |
| Save draft | Persists ONE policy YAML document via existing v0.1 flow. New section adds ~3–8KB to the YAML blob. SQLite write < 5ms. |

| Validation cost | `re.compile()` over user patterns: O(n × pattern_length) — typical: <30 patterns × <100 chars ≈ <3000 chars total → sub-millisecond on CPython. Performed server-side at draft-save time only (NOT on every render). |
| No polling on `/policy` | Editor pages are not polled (matches v0.1 + Phase 4). |
| No new dependencies | **No chart library, no JS framework, no toast library.** All UI = Jinja + Tailwind + HTMX (already loaded). No HTMX needed in this card — entire form re-renders on standard POST. |
| Tailwind CDN | Continues to use `cdn.tailwindcss.com` runtime — no build step. |
| Backward-compat read path | If the YAML blob on disk has no `prompt_injection` section (pre-Phase-5 deployments), Pydantic defaults populate it (`enabled=True`, `severity=warn`, empty lists, LlamaGuard `enabled=False`). Renderer shows defaults — zero additional latency. |
