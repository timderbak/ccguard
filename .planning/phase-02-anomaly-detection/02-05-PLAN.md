<objective>
Ship the two main user-facing surfaces of Phase 2:
1. /anomalies — fleet-wide matrix table (rows = machines, columns = 4 metrics), each cell a CSS sparkline + last value. HTMX-polled every 60s.
2. /anomalies/{machine_id}/{metric} — read-only drill-down: back link, heading, baseline summary strip, 14-day CSS bar chart with baseline band overlay and outlier dots, list of recent matching Findings.

All UI strings RU verbatim per UI-SPEC. CSS-only charts (no JS). Reuses Phase 1 Jinja+HTMX+Tailwind conventions.

Purpose: This is where admins investigate an anomaly. The matrix gives "where is the problem?", the drill-down gives "what does the trend look like and which findings have fired?".

Output: 3 new templates + 3 new routes wired into the existing router.
</objective>

  <action>
    Add three handlers, using the same auth/session/templates idiom established by plan 02-04 (read 02-04-SUMMARY.md for the exact symbol names).

    1. GET /anomalies — renders anomalies_feed.html with {request}. The page itself does NOT pre-fetch matrix data; the wrapping HTMX div fetches /_partials/anomalies/matrix on load.

    2. GET /_partials/anomalies/matrix — builds matrix view-model:
       - Query all Machine rows ordered by last_seen DESC.
       - For each machine, for each metric in ALL_METRICS:
         - SELECT MachineBaseline row by (machine_id, metric). If none, cell view-model is: {warmup: True, points: [], last_value: None, is_outlier: False}.
         - Otherwise: parse recent_points_json into list[float]; last_value = points[-1] if points else None; max_val = max(points) if points else 1; compute height_pct per point as (p/max_val)*100; is_outlier = baseline.baseline_ready and stdev > 0 and abs(last_value - mean) > 3*stdev. View-model: {warmup: not baseline.baseline_ready, points: [{value, height_pct, label}...], last_value, is_outlier}.
       - Render components/_anomalies_matrix.html with {request, machines: [{id, cells: {metric: cell_vm}}, ...]}.

    3. GET /anomalies/{machine_id}/{metric} — 404 if metric not in VALID_METRICS. Otherwise:
       - Load MachineBaseline by (machine_id, metric); may be None (no tick yet).
       - Build points view-model: if baseline exists parse recent_points_json (14 floats), else 14 zeros. Compute height_pct per point with min-height per UI-SPEC (>=2px if value>0). is_outlier per point = baseline_ready and stdev>0 and abs(value - mean) > 3*stdev.
       - Compute band overlay: max_val = max(points) or 1; band_bottom_pct = max(0, ((mean - 3*stdev) / max_val) * 100); band_height_pct = min(100 - band_bottom_pct, (6*stdev / max_val) * 100). If baseline_ready=False, set band_visible=False.
       - Query recent FindingRecord rows where machine_id == m AND rule_id == anomaly.{metric} ORDER BY discovered_at DESC LIMIT 50. Parse payload_json for each row to expose observed_value and sigma_distance in the template.
       - Render anomaly_detail.html with {request, machine_id, metric, baseline (or None), points, band_visible, band_bottom_pct, band_height_pct, findings, baseline_ready}.
       - For label dates: 14 daily anchors `[(today_utc - timedelta(days=13-i)) for i in range(14)]` rendered as ISO date strings.
  </action>
