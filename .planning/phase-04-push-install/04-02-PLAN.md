<objective>
Wire the admin UI for the new «Обязательные» tab on /policy: a shared tab
strip, a new page `/policy/mandatory` with four `<details>` section cards, an
HTMX row-add endpoint, and an extended `POST /policy/draft` parser that
accepts indexed form fields for all four new sections. Russian UI copy is
locked verbatim per 04-UI-SPEC.md.

Purpose: Phase 4 admin can only describe required artifacts through this UI.
The agent (plan 03) and API (`/api/v1/policy`) consume the resulting Policy
revision. Without this plan, plans 03–06 have no admin-authored input to
operate on.

Locked decisions applied:
- D-5: `required_skills[].content` = full file (frontmatter + body) in a single
  textarea — no separate frontmatter editor
- D-6: MCP args/env editors are single inline JSON textareas (args as
  comma-separated text per UI-SPEC row template, env as JSON string) — Monaco
  is overkill for v0.2
- D-7: managed MCP entries are identified by `_managed_by: "ccguard"` — the
  UI stores this on the server side automatically when persisting MCP entries
  from `required_mcp_servers` into the policy YAML; admins never see the
  field in the editor (used by agent in plan 03 for merge logic)

Output: New page, 4 section partials, 4 row partials, shared tab-strip
partial, route handlers, form parser additions, and integration tests proving
end-to-end CRUD plus the published-revision round-trip.
</objective>
