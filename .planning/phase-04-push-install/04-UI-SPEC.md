# Phase 4 — UI Design Contract: Push-Install (Mandatory Tab)

> Visual and interaction contract for the new «Обязательные» tab on `/policy` (4 editor sections — `required_mcp_servers`, `required_skills`, `required_agents`, `managed_claude_md_blocks`), the new audit event source on `/audit` (`policy_apply`), and the existing Draft → Publish → History flow extended to cover the new sections. Generated by gsd-ui-researcher.
>
> **Constraint:** Extends the same Russian-language admin UI (Jinja2 + HTMX + Tailwind CDN) as Phases 1–3. **Reuses Phase 1+2+3 design tokens verbatim** — no new colors, no new typography, no new font sizes, **no client-side JavaScript** beyond the HTMX runtime + native `confirm()` already used in `policy_editor.html`. shadcn is **not applicable** (server-rendered Jinja, no React). Editors are plain `<textarea>` per CONTEXT.md decision (Monaco overkill for v0.2).

## Information Architecture

### Navigation

**No new sidebar entries.** The Mandatory editors live inside the existing `/policy` page as a **tab strip**, and the apply events surface inside the existing `/audit` page as a new `event_source` filter value.

Final sidebar order: `Обзор / Машины / Находки / Аудит / Аномалии / Политика / История / Настройки` — unchanged from Phase 2.

### Page structure: `/policy` (extension — tab strip)

The single existing form is split into **two tabs** above the form, using server-rendered tab links (no JS — each tab is its own URL):

```
/policy            → tab «Правила»       (existing v0.1 form: MCP / Network / Commands / Skills / Hooks / Agents / Env)
/policy/mandatory  → tab «Обязательные»  (NEW: 4 editor cards)
```

Tab strip markup, rendered above `<h2>Редактор политики</h2>` on both pages:

```html
<nav class="flex gap-2 border-b border-slate-200 mb-6 text-sm" aria-label="Разделы политики">
  <a href="/policy"
     class="px-3 py-2 -mb-px border-b-2

            {% else %}border-transparent text-slate-500 hover:text-slate-700{% endif %}">
```

Both tabs share the **same revision header** (`Текущая ревизия / Ревизия черновика / · не сохранено`) and the **same sticky bottom action bar** (Сохранить черновик / Опубликовать). Publishing from either tab bumps the same revision — there is one policy document with both rule-sections and mandatory-sections.

Top-down vertical stack inside `<main class="flex-1 p-8">`:

1. **Tab strip** (above heading) — see markup above
2. **Page heading** — `<h2 class="text-2xl font-semibold mb-2">Редактор политики</h2>` (same string as `/policy` — tabs are sub-sections of one editor)

### URL contract

| Route | Method | Effect / Returns |
|-------|--------|------------------|
| `GET /policy` | GET | full `policy_editor.html` with `active_tab='rules'` — existing, unchanged |

## Interaction & State

### Tab switching

- Native `<a href="/policy/mandatory">` navigation — full page reload. **No JS tab switcher.** Each tab is its own URL and gets its own browser-history entry.
- Active tab is rendered server-side from the URL — no client state.
- Unsaved draft warning: not implemented in v0.2 (out of scope; matches existing v0.1 — switching pages with unsaved form data discards it. Users are trained by the existing «· не сохранено» indicator and the dual-submit bottom bar.).

### Adding / removing list items in editors

Each of the 4 sections is a **dynamic list**. Add/remove rows uses HTMX server-rendered partials (no client-side templating):

```html
{# Add button — fetches one empty row partial and appends it #}
<button type="button"
        hx-get="/policy/mandatory/_row?section=required_mcp_servers&i={{ next_index }}"
        hx-target="#mcp-rows"
        hx-swap="beforeend"
        class="text-sm text-slate-700 hover:underline">
  + добавить сервер
</button>
```

```html
{# Remove button — inside each row, removes the closest <div class="row"> #}
<button type="button"
        hx-on:click="this.closest('.policy-row').remove()"
        class="text-xs text-red-600 hover:underline">
  удалить
</button>
```

**Note on `hx-on:click`:** HTMX 1.9+ supports the `hx-on:` attribute as a sanctioned inline event handler that does **not** require any external script asset (only the HTMX runtime already loaded by `base.html`). This stays within the no-new-JS constraint, matching the precedent of `onsubmit="return confirm(...)"` from `policy_editor.html` line 27.

Alternative (executor discretion): if HTMX's `hx-on:` is judged out of taste, fall back to `<button hx-delete="/policy/mandatory/_row" hx-target="closest .policy-row" hx-swap="delete">` — server returns empty 200. UI-SPEC accepts either.

New server routes for row-add (listed under **URL contract** below):

| Route | Returns |
|-------|---------|
| `GET /policy/mandatory/_row?section={section}&i={index}` | Single empty row partial (`components/_mandatory_row_{section}.html`) for HTMX append |

### Saving draft

Single combined `<form method="POST" action="/policy/draft">` containing ALL 4 sections (one form per page — same as `/policy` rules tab). Hidden `<input type="hidden" name="tab" value="mandatory">` so the server returns the user to the correct tab on validation error.

- Server validates each section with Pydantic v2 (per CONTEXT.md decision).
- **Validation error path:** re-render `/policy/mandatory` with the error notice atop the offending section card, draft content preserved in form values. Notice copy: see Copywriting Contract.
- **Success path:** 303 redirect to `/policy/mandatory` (preserves tab); `has_draft` flips to `True`; revision header shows `· не сохранено`.

### Publishing

Identical to v0.1: `<button type="submit" formaction="/policy/publish">Опубликовать</button>` with `onclick="return confirm('Опубликовать черновик на все машины?')"`. After publish: 303 to `/policy/history`. New revision contains the new sections (verified in `policy_history.html` diff view via existing diff renderer — no UI changes there).

### Audit filter

Selecting «События политики» in the existing `event_source` `<select>` and submitting the existing «Фильтр» button reloads `/audit?event_source=policy_apply` with:
- Existing columns preserved (Когда / Машина / Источник / details)
- One new column inserted at the rightmost position: **«Результат»** — renders one of two pills:
  - `success` → `<span class="inline-block rounded-full px-2 py-0.5 text-xs font-semibold bg-emerald-600 text-white">success</span>`
  - `rollback` → `<span class="inline-block rounded-full px-2 py-0.5 text-xs font-semibold bg-red-600 text-white">rollback</span>`
- `details` cell shows: for `success` — `applied={n}, snapshot={snapshot_id[:8]}`; for `rollback` — `reason={reason}, failed_file={failed_file}, snapshot={snapshot_id[:8]}` (font-mono, truncated `max-w-md`)

### States

| State | Trigger | UI |
|-------|---------|-----|
| Default (with draft) | `has_draft=True`, sections populated | Revision header shows `· не сохранено`; all 4 cards open (`<details open>`); rows populated from draft |
| Default (no draft) | `has_draft=False` | Cards open; rows populated from latest published revision; no «не сохранено» badge |
| Empty section | zero items in a section | Inside the card: `<p class="text-sm text-slate-400 py-4 text-center">Записей нет.</p>` above the "+ добавить" button |
| Cold-start (no policy yet) | first ever load — server seeds `revision=1` empty draft | All 4 sections render with empty state |
| Validation error | Pydantic rejects one or more sections | Re-render `/policy/mandatory`; error notice in `text-red-600` atop the offending card; form values preserved |
| Loading | initial page render | Server-rendered HTML; no spinners; HTMX row-add swaps silently |
| Server 5xx on row-add | HTMX `_row` GET fails | HTMX default: keep last successful render. No toast. |
| Publish in flight | between submit and 303 | Browser default — no custom indicator |
| Audit filter — empty results | no `policy_apply` events yet | Existing audit empty-row pattern: `<tr><td colspan="N" class="py-6 text-center text-slate-400">Событий нет.</td></tr>` |
| Audit row — rollback | `event.result == 'rollback'` | Red pill in «Результат» column; `details` cell shows reason in `text-red-600` (only the word `reason=` highlighted; the reason value stays `font-mono text-slate-700`) |

### Confirmations / destructive actions

| Action | Mechanism | Confirmation copy (locked) |
|--------|-----------|----------------------------|
| Publish draft (existing — unchanged) | `onclick="return confirm(...)"` | `Опубликовать черновик на все машины?` |
| Rollback to revision (existing — unchanged) | `onclick="return confirm(...)"` | `Создать черновик из ревизии {revision}?` |
| Remove row (new) | none — clicking «удалить» removes immediately, but the row is **not persisted** until the user clicks «Сохранить черновик». No data loss until save. | — |
| Add row (new) | none — non-destructive | — |
| Save draft (new sections) | none — non-destructive, reversible via rollback in history | — |

**Note:** Per CONTEXT.md the actual policy apply on endpoints is **pull-based** (agent fetches at sync interval). There is NO «Применить ко всем машинам» button. The Опубликовать button only writes the new revision to the server DB; agents pick it up via existing ETag-cached `/api/v1/policy`. UI-SPEC must NOT introduce a push-button — explicitly out of scope per CONTEXT.md «Specific Ideas».

## Accessibility

| Concern | Implementation |
|---------|----------------|
| Semantic structure | `<main>`, `<h2>` page title, `<nav aria-label="Разделы политики">` for tab strip, `<details>`/`<summary>` for collapsible cards, `<form>` wrapping all sections, `<label>` wrapping every form control (visible text) |
| Tab semantics | Plain `<a>` links — not ARIA `role="tab"`. Each tab is its own URL; full reload. Active state communicated via `font-semibold` + underline color (not color alone — weight is the primary signal). No `aria-selected` needed because the active page is implicit from the URL. |
| Form labels | Every input/textarea has a wrapping `<label>` with a `<span class="text-xs text-slate-500">` field name. No `placeholder`-only labels. |
| Focus states | All `<input>`/`<textarea>`: rely on Tailwind defaults; executor adds `focus:ring-2 focus:ring-slate-400 focus:border-slate-500` if missing (matches Phase 3 pattern). All `<button>`: `focus-visible:ring-2 focus-visible:ring-slate-500`. |
| Keyboard navigation | Tab order: tab links → section card summaries (focusable via `<summary>`) → inside each card: name → command/type → args/content → удалить → next row → … → «+ добавить» button → next section card → … → «Сохранить черновик» → «Опубликовать». Native — no `tabindex` overrides. |
| HTMX row-add | New row appears in the DOM via `hx-swap="beforeend"`. No focus management beyond browser default (executor may add `hx-on::after-swap="this.previousElementSibling.querySelector('input').focus()"` — out of scope for v0.2, deferred). |
| HTMX row-remove | `hx-on:click="this.closest('.policy-row').remove()"` removes the DOM node — screen readers will not re-announce; acceptable for admin tool. No undo affordance. |
| Color contrast | `bg-emerald-600` on white ≥ 4.5:1 ✓; `bg-red-600` on white ≥ 4.5:1 ✓ (both verified in Phase 1–3); `text-red-600` for error notices ≥ 4.5:1 on white ✓; `text-slate-500` for helper text ≥ 4.5:1 ✓. |
| Result pill not color-only | The pill always carries visible text (`success` / `rollback`). Color is reinforcement. |
| Confirm dialog | Browser-native `confirm()` — fully keyboard-accessible by default. |
| Section card collapse | `<details>` is natively keyboard-accessible (Enter on `<summary>` toggles). |
| Active tab indication not color-only | Active tab combines: `font-semibold` (weight) + `text-slate-900` (color) + `border-slate-900` bottom underline (geometry) — three reinforcing signals. |

## Performance

| Concern | Decision |
|---------|----------|
| `/policy/mandatory` initial render | One query for current draft (existing v0.1 policy table — now contains 4 extra YAML sections). Render cost: ≤ ~50 rows total across all 4 sections in typical org → ≤ ~50 textareas × ~2KB markup ≈ 100KB. Acceptable. Cards `<details open>` by default — no reflow on user interaction. |
| Row-add HTMX endpoint | `GET /policy/mandatory/_row` — single Jinja partial render, no DB query (server holds `next_index` in a counter; client passes `i` in URL). Sub-millisecond. |
| Save draft | Persists ONE policy YAML document; existing v0.1 flow. New sections add ~5–20KB to the YAML blob. SQLite write < 5ms. |
| ETag-cache invalidation | Automatic via existing content-hash mechanism (CONTEXT.md — code_context). No new code path. |
| `/audit?event_source=policy_apply` query | Indexed scan on `event_source` column. Existing v0.1 audit query pattern with new `WHERE event_source='policy_apply'` predicate. Planner ensures index exists (recommended in v0.1 already). |
| No polling on `/policy/mandatory` | Editor pages are not polled (matches v0.1). User refreshes manually. |
| No polling on `/audit` policy-apply filter | Audit feed is not polled in v0.1 — unchanged in this phase. |
| No new dependencies | **No chart library, no JS framework, no toast library.** All UI = Jinja + Tailwind + HTMX (already loaded). Uses HTMX 1.9+ native `hx-on:click` (sanctioned, no script asset). |
| Tailwind CDN | Continues to use `cdn.tailwindcss.com` runtime — no build step. |
