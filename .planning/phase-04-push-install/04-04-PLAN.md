<objective>
Wire the agent CLI sync command to the apply pipeline and extend the server's
POST /api/v1/audit endpoint to accept and persist `policy_apply` events. The
integration is best-effort: a failure anywhere in the apply or audit-post
chain must never crash the CLI's normal sync flow.

Purpose: Plan 03 built the apply engine in isolation. This plan integrates
it into the user-facing sync command and gives the server the ability to
record what happened. Plan 05 (UI extension) and plan 06 (e2e tests) both
depend on the policy_apply event existing on the wire.

Locked decisions applied:
- D-1 schema_version=1: backward-compat for v0.1 agents — the new audit
  endpoint branch is purely additive; v0.1 agents that never post
  `event_source='policy_apply'` continue to work unchanged
- D-7 `_managed_by: "ccguard"` is set by the server during policy publish
  (plan 02) and consumed by the agent during merge (plan 03); the CLI just
  passes the policy through

Output: Updated sync flow, new audit endpoint branch, payload schema, and
two integration tests covering both the CLI happy path and the server-side
endpoint accept/persist behavior.
</objective>

    - If the audit POST itself fails (server 5xx, network error), the CLI sync command STILL exits 0 — apply result is logged at WARNING but not raised
    - If the policy response is from a v0.1 server (no required_* sections), apply() runs against an empty policy and returns success with applied_count=0; no audit POST is sent for the no-op case (avoid noise — only POST when at least one target file was written OR a rollback occurred)
    - Repeated sync calls are idempotent: re-running with the same policy produces success with the same applied_count and does not corrupt files
    - Machine_id used in the audit POST comes from existing agent/machine_id.py (no duplication)
    - Sync flow ordering: fetch policy → existing v0.1 inventory upload → push_install.apply → audit POST (so a failing apply does not block inventory upload, and vice versa)
