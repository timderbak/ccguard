<objective>
Lay the data + schema foundation for the LLM content scanner: extend the existing
`Severity` Literal with a fourth value `critical`, add two new SQLModel tables
(`ScanResult`, `LLMCallLog`), seed the existing `SettingsRecord` KV table with the
two new admin keys (`llm_scanner_enabled`, `daily_call_budget`), and register the
new tables with `create_all` plus composite indexes.

Purpose: Without this layer, no later plan can store a scan, log a call, or read
admin settings. Locked decisions D-01 (severity=critical additive) and D-04
(SettingsRecord KV) from CONTEXT.md drive the choices here.

Output: Updated `Severity` Literal, new ORM models, new indexes, regex in
findings API, and two unit-test files proving the schema works end-to-end at
the model layer.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| admin browser → server | cookie-authenticated admin can change `daily_call_budget` and toggle scanner |
| db disk → restore | offline DB snapshot may leak file_hash + risk_score + rationale (but never raw content) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | SettingsRecord rows for llm_scanner_enabled / daily_call_budget | mitigate | only admin cookie auth (existing Phase 1 pattern) can POST /admin/llm-settings; reuse existing CSRF middleware |
| T-03-02 | Information disclosure | ScanResult.rationale | accept | rationale is model-generated summary, capped at 500 chars; never echoes raw secrets because content is masked before send (Plan 04 enforces); admin-only read |
| T-03-03 | Denial of service | LLMCallLog row growth | mitigate | bounded by daily_call_budget (default 100/day) enforced in Plan 03; admin can audit via Settings UI |
</threat_model>
