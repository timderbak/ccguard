<objective>
Wire scan_service to HTTP and the agent. Server gets a new router with two
endpoints (one for the agent to call with content, one for the agent to query
whether scanning is enabled). Agent gets a masking helper, a content-collection
module, and a CLI integration point.

Locked decisions:
- D-02 one-pass protocol: agent sends content+hash in a single POST; server doesn't ask for content separately
- Privacy: server NEVER persists raw content — only hash + score + category
- Reuse v0.1 secret-masking regex from existing agent masking pattern
- Backward compat: scanner-config returns enabled=false if Settings.llm_scanner_enabled=false OR ANTHROPIC_API_KEY missing — old agents that don't call this endpoint simply skip scanning

Output: new schemas, new server router mounted in main.py, new agent module +
CLI hook, 8+ tests (endpoint + masking + agent payload shape).
</objective>

| server → DB | only hash + score + category + rationale persisted; raw content never touches disk |
| server logs | structured logging excludes content bytes |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-10 | Information disclosure | content in transit | mitigate | TLS in deployment + agent-side mask_secrets before send (this plan, Task 2) + size cap |
| T-03-11 | Information disclosure | content in server logs | mitigate | structured logger config excludes content field; explicit test asserts no content in captured logs |
| T-03-12 | Spoofing | unauthenticated scan POST | mitigate | reuse existing X-CCGuard-Token agent auth (Phase 1 pattern); unauthorized requests rejected with 401 |
| T-03-13 | DoS | oversized batch | mitigate | max 50 items per request; 1MB hard cap per item; over-cap items rejected without LLM call |
