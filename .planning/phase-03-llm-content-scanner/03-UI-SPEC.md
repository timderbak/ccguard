# Phase 3 — UI Design Contract: LLM Content Scanner

> Visual and interaction contract for the `/findings` page extensions (risk_score column, badge, per-row re-scan, filter for `rule_id LIKE 'llm.scan.*'`) and the `/settings` page extensions («LLM-сканер» section: toggle, daily_call_budget input, usage counter, last 10 ScanResult list, "Пересканировать всё" button). Generated by gsd-ui-researcher.
>
> **Constraint:** Extends the same Russian-language admin UI (Jinja2 + HTMX + Tailwind CDN) as Phases 1–2. **Reuses Phase 1+2 design tokens verbatim** — no new colors, no new typography, no new font sizes, **no client-side JavaScript** beyond the HTMX runtime already loaded by `base.html`. shadcn is **not applicable** (server-rendered Jinja, no React).

## Information Architecture

### Navigation

**No new sidebar entries.** All Phase 3 UI lives inside `/findings` and `/settings` — both pages already exist in v0.1 and are linked from the sidebar.

### Page structure: `/findings` (extension)

Existing structure preserved: filter bar (card 1) + findings table (card 2). Two additive changes:

1. **Filter bar gains one new control** — a `<select name="scope">` between the existing `severity` and `rule_id` controls (exact position: immediately after `severity`).
2. **Findings table gains two new columns** — `risk_score` (after the existing `severity` column) and `Действия` (last column, rightmost, holding the per-row "Пересканировать" button).

Both changes are additive: rows whose `rule_id` does **not** start with `llm.scan.` render an em-dash (`—`) in the new cells. No data is hidden from existing findings.

### Page structure: `/settings` (extension)

Append one new section card below the existing settings cards inside `<main class="flex-1 p-8">`:

```html
<section class="bg-white rounded-lg shadow p-4 mb-4">
  <h3 class="text-sm font-semibold text-slate-700 mb-3">LLM-сканер</h3>
  {# 1. Toggle row #}
  {# 2. Budget input row #}
  {# 3. Usage counter (HTMX-polled) #}
  {# 4. Last 10 ScanResult list #}
  {# 5. "Пересканировать всё" button #}
</section>
```

Vertical order inside the card (top-to-bottom):
1. Toggle row (enabled/disabled)
2. Daily call budget input
3. Usage counter strip
4. Last 10 ScanResult list
5. Global re-scan button

Each sub-block separated by `mb-4` (16px). The toggle and budget input share one `<form method="POST" action="/admin/llm-settings">`. The global re-scan button is a separate `<form method="POST" action="/admin/scan/rescan-all">`.

### URL contract

**`/findings` query params (additive — preserves Phase 1+2 contract):**

| Param | Type | Allowed values | Default |
|-------|------|----------------|---------|
| `machine_id` | string (fragment match) | any | empty (all) — existing |
| `severity` | enum | `info`, `warn`, `critical`, `` | empty — existing |
| `rule_id` | string (fragment match) | any | empty — existing |
| `scope` | enum | `all`, `llm`, `non_llm` | `all` — **new in Phase 3** |

Backend treatment of `scope`:
- `all` (or missing) → no filter on rule_id beyond existing `rule_id` param
- `llm` → adds `AND rule_id LIKE 'llm.scan.%'`
- `non_llm` → adds `AND rule_id NOT LIKE 'llm.scan.%'`

The `scope` filter composes with the existing `rule_id` fragment-match filter — both are ANDed.

**New admin routes (server-side, listed for completeness):**

| Route | Returns / Effect |
|-------|------------------|
| `POST /admin/llm-settings` | persist `enabled` + `daily_call_budget`; redirect 303 to `/settings` |
| `POST /admin/scan/{file_hash}/rescan` | invalidate cache for `file_hash`, immediate Anthropic call (subject to budget); returns the **single rescanned `<tr>` partial** (`components/_finding_row.html`) — HTMX swaps the originating `<tr>` |
| `POST /admin/scan/rescan-all` | bulk invalidate all `ScanResult` rows; enqueue immediate rescan; redirect 303 to `/settings` |
| `GET /_partials/settings/llm-usage` | `components/_llm_usage_counter.html` — usage strip (polled every 30s) |

## Interaction & State

### Filter submission (`/findings`)

- Existing submit button + reset link unchanged.
- New `<select name="scope">` is a sibling of the existing filter inputs, included in the same `<form method="GET" action="/findings">`. Submission via the existing "Фильтр" button reloads the page with the chosen scope in the URL.
- Selected `scope` is preserved across submissions via `selected` attribute on the matching `<option>` (same pattern as existing `severity` select).

### Per-row re-scan (`/findings`)

The "Действия" cell contains a button **only** for rows whose `rule_id LIKE 'llm.scan.%'` AND whose `details.file_hash` is non-empty. Other rows show `—`.

```html
<form hx-post="/admin/scan/{{ finding.details.file_hash }}/rescan"
      hx-target="closest tr"
      hx-swap="outerHTML"
      hx-confirm="Пересканировать этот файл? Списание из дневного бюджета."
      class="inline">
  <button type="submit"
          class="text-sm text-slate-700 hover:text-slate-900 hover:underline">
    Пересканировать
  </button>
</form>
```

- Confirmation: native `hx-confirm` dialog (no custom modal — keeps CSS-only constraint).
- Success path: server returns the rescanned row's `<tr>` HTML; HTMX swaps it in place. No page reload, no flash message.
- Budget-exhausted path: server returns HTTP 200 with the **same `<tr>`** rendered but with an inline `<span class="text-xs text-amber-600 ml-2">бюджет исчерпан</span>` after the score value. The original row state is preserved; only the inline notice is added. Notice persists until next manual filter reload.
- HTTP 500 path: HTMX leaves the original `<tr>` intact (HTMX default `hx-swap-oob` behavior). No toast.
- Disabled-scanner path: button still renders, but clicking returns the row with inline `<span class="text-xs text-slate-500 ml-2">сканер выключен</span>`.

### Settings toggle + budget save

Single combined `<form method="POST" action="/admin/llm-settings">` containing:
- `<input type="checkbox" name="enabled">` — checked iff scanner is enabled
- `<input type="number" name="daily_call_budget" min="0" max="10000" step="1">` — integer
- `<button type="submit">Сохранить</button>`

No client-side validation. Server validates: `0 ≤ daily_call_budget ≤ 10000`. On invalid input → re-render `/settings` with error notice atop the LLM section. On success → 303 to `/settings`.

### Usage counter (HTMX-polled)

```html
<div class="text-sm"
     hx-get="/_partials/settings/llm-usage"
     hx-trigger="every 30s">
  {% include "components/_llm_usage_counter.html" %}
</div>
```

Polling cadence: **30s** (matches budget granularity expectations from CONTEXT.md). The partial renders a single line:

```
Использовано: {used}/{budget} calls сегодня • ${cost_dollars}
```

If `enabled=False`: line reads `Сканер выключен.` and polling continues (cheap idempotent GET, no-op server-side).

### Global "Пересканировать всё" button

```html
<form method="POST" action="/admin/scan/rescan-all"
      onsubmit="return confirm('Пересканировать ВСЕ файлы? Может превысить дневной бюджет.');">
  <button type="submit"
          class="bg-slate-900 text-white text-sm rounded px-4 py-2">
    Пересканировать всё
  </button>
</form>
```

**Note on `onsubmit`:** This is the *only* inline JS in the entire Phase 3 contract. Native `confirm()` is permitted by the CSS-only constraint because it does not require a script asset — it's a built-in browser API invoked from an HTML attribute. If the executor objects, fall back to a two-step confirmation page (`POST /admin/scan/rescan-all/confirm` → confirmation form → `POST /admin/scan/rescan-all/commit`). UI-SPEC accepts either; default is inline `confirm()`.

### States

| State | Trigger | UI |
|-------|---------|-----|
| Default (with data) | findings exist; scanner enabled; budget remaining | All cells populated; usage counter shows e.g. `Использовано: 23/100 calls сегодня • $0.12` |
| Empty — findings table | zero findings match filter | Existing empty state preserved: `<tr><td colspan="N" class="py-6 text-center text-slate-400">Находок нет.</td></tr>` (N updated to include the 2 new columns) |
| Empty — last 10 ScanResult | zero scans performed ever | `<p class="text-sm text-slate-400 py-4 text-center">Сканов ещё не было.</p>` |
| Scanner disabled | `Settings.llm_scanner_enabled=False` | Toggle unchecked; budget input still editable; usage strip reads `Сканер выключен.`; per-row buttons render but return "сканер выключен" notice on click |
| Budget exhausted | `used >= budget` for today | Usage strip shows count in `text-red-600`; per-row buttons return "бюджет исчерпан" notice on click; cache hits continue to work transparently |
| Warm-up / no API key | `ANTHROPIC_API_KEY` empty at server startup | LLM-сканер section renders with toggle disabled (`<input disabled>`) + notice `<p class="text-sm text-amber-600">ANTHROPIC_API_KEY не задан в окружении сервера. Сканер отключён.</p>` above the toggle |
| Loading | initial page render | Server-rendered HTML; HTMX swaps silently (no skeleton) |
| Error (server 5xx on partial) | HTMX poll fails | HTMX default: keep last successful render. No toast. |
| Validation error on save | `daily_call_budget` out of range | Re-render `/settings` with notice atop LLM section: `<p class="text-sm text-red-600 mb-3">Бюджет должен быть целым числом от 0 до 10000.</p>` |

### Confirmations / destructive actions

| Action | Mechanism | Confirmation copy |
|--------|-----------|-------------------|
| Per-row re-scan | `hx-confirm` (HTMX built-in) | `Пересканировать этот файл? Списание из дневного бюджета.` |
| Global re-scan | inline `onsubmit="return confirm(...)"` | `Пересканировать ВСЕ файлы? Может превысить дневной бюджет.` |
| Toggle save | none (non-destructive, immediately reversible) | — |
| Budget save | none (non-destructive) | — |

### Risk badge spec (CSS-only, no JS)

A small pill-shaped badge showing the integer score, color-coded by range.

```html
{% if finding.details.risk_score is not none %}
  {% set s = finding.details.risk_score %}
  <span class="inline-block rounded-full px-2 py-0.5 text-xs font-semibold
               {% if s < 30 %}bg-emerald-600 text-white
               {% elif s <= 70 %}bg-amber-600 text-white
               {% else %}bg-red-600 text-white{% endif %}"
        title="risk_score={{ s }} • {{ finding.details.category }}">
    {{ s }}
  </span>
  <span class="text-xs text-slate-500 ml-1 font-mono">{{ finding.details.category }}</span>
{% else %}
  <span class="text-slate-400">—</span>
{% endif %}
```

**Sizing rules:**
- Badge: `px-2 py-0.5` (8px horizontal, 2px vertical) + `rounded-full` + `text-xs font-semibold`
- Category label sits to the right of the badge, `text-slate-500 font-mono text-xs`, separated by `ml-1` (4px)
- Hover tooltip via native `title` attribute exposes the full score + category for screen readers

## Performance

| Concern | Decision |
|---------|----------|
| New findings columns | Zero additional queries — risk_score + category live in `Finding.details` JSON (already SELECTed in v0.1). Render cost: ~2 extra cells × ~80 bytes per row × 200 rows = ~32KB additional payload. Acceptable. |
| Scope filter | Server-side `AND rule_id LIKE 'llm.scan.%'` — index on `rule_id` recommended (planner to confirm; existing v0.1 already filters on rule_id fragment match). |
| Settings page render | Adds 1 extra query for `ScanResult ORDER BY scanned_at DESC LIMIT 10` + 1 query for today's `LLMCallLog` aggregate (sum input/output tokens). Both indexed on `scanned_at` / `ts` — sub-millisecond on SQLite. |
| Usage counter poll | **30s** interval. Endpoint returns ~200 bytes. Aggregated daily-counter cached in memory for 5s (planner discretion). |
| Per-row re-scan response | Synchronous Anthropic API call (1–3s p95 for Haiku). Server holds `asyncio.Lock` per CONTEXT.md (sequential, no concurrency). HTMX shows no spinner — acceptable for admin tool. If executor wants a spinner: add `hx-indicator` pointing at an `<svg class="htmx-indicator">` (out of scope for this UI-SPEC; no JS required for indicator). |
| Global re-scan response | Server enqueues bulk job + returns 303 immediately. Progress is observable via the usage counter (which increments as scans complete) — no dedicated progress UI in v0.2. |
| No new dependencies | **No chart library, no JS framework, no toast library.** All UI = Jinja + Tailwind + HTMX (already loaded). The single use of inline `onsubmit="return confirm(...)"` does not constitute a new dependency — it's a browser built-in. |
| Tailwind CDN | Continues to use `cdn.tailwindcss.com` runtime — no build step. |
| HTMX include scope | `_partials/settings/llm-usage` poll is global — does not include any form values, so no `hx-include` directive. |
