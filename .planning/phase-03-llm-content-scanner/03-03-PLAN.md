<objective>
Glue layer: take a (content, file_path, scope) input, decide whether to hit the
cache or call Anthropic via LLMClient, enforce the daily budget, persist the
result, and emit a Finding when risk crosses the threshold. This service is
the single entry point the HTTP layer (Plan 04) calls.

Locked decisions:
- D-03: re-scan-all is an APScheduler one-shot job (this plan exposes the per-file primitives; the scheduler hook is in Plan 05)
- D-04: emit threshold = `risk_score >= 30`, rule_id format `llm.scan.{category}`
- D-02: one-pass — server receives content from agent in one call; we never ask the agent for more

Output: scan_service module with `scan_file`, `rescan_file`, `get_daily_usage`,
plus tests covering cache hit/miss, budget exhaustion, lock serialization,
severity mapping, and finding emit.
</objective>

  <name>Task 1: Implement scan_service with cache, budget gate, async lock</name>
  <files>src/ccguard/server/services/scan_service.py, tests/unit/test_scan_service.py</files>
  <behavior>
    - `class ScanService` holds an `asyncio.Lock` instance attribute and references to LLMClient + DB session factory
    - `scan_file(content: str, file_path: str, scope: Literal["agent","skill"], force_rescan: bool = False) -> ScanResult`:
        1. Compute file_hash = sha256(content.encode("utf-8")).hexdigest()
        2. If not force_rescan: query ScanResult by file_hash; if row exists and ttl_expires_at > utcnow → return cached row, do NOT call LLM, do NOT decrement budget
        3. Check budget: today's LLMCallLog count (UTC day) vs SettingsRecord daily_call_budget; if exhausted → raise BudgetExhaustedError
        4. Check scanner_enabled SettingsRecord; if false → raise ScannerDisabledError
        5. Acquire self._lock, call LLMClient.scan_content, release lock
        6. Insert LLMCallLog row with ts/file_hash/model/tokens/cost
        7. UPSERT ScanResult by file_hash with risk_score, category, rationale, scanned_at=utcnow, ttl_expires_at=utcnow+30d, model
        8. If risk_score >= 30: emit Finding via finding_service helper (rule_id=f"llm.scan.{category}", severity per mapping, details JSON includes file_hash + risk_score + category + scope + file_path)
        9. Return the persisted ScanResult
    - `rescan_file(file_hash: str)`: looks up existing ScanResult by file_hash to recover file_path + scope; we DO NOT store content. Re-scan therefore requires the agent to resend content. **Architectural note for executor:** since server never stores content (D-privacy locked), `rescan_file` cannot itself trigger a fresh Anthropic call. Instead, `rescan_file(file_hash)` marks the cache as expired (`ttl_expires_at = utcnow - 1s`) and returns a sentinel `RescanQueued` — the agent's next inventory cycle will resend content and the cache miss path will perform the actual scan. The "Пересканировать всё" button maps to bulk expiry of all ScanResult rows.
    - `get_daily_usage() -> dict`: returns `{"used": int, "budget": int, "cost_cents": int, "enabled": bool}` aggregating today's LLMCallLog rows and reading SettingsRecord
    - Severity mapping: score<30→"info", 30<=score<=70→"warn", score>70→"critical" (matches UI-SPEC badge boundaries; per D-01 the new critical literal is in scope)
    - All DB writes inside a single transaction per scan_file call (atomic: log + ScanResult upsert + Finding emit either all commit or all rollback)
    - asyncio.Lock proven by test: launch two scan_file coroutines concurrently with a mock LLMClient that records call entry timestamps; assert they do NOT overlap
  </behavior>
